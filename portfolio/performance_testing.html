Performance Testing

API and Browser Load Testing

Overview

In 2023, mabl developed and launched a new product to allow users to create and run performance tests. I led the research, ideation, and design of the new product experience that result in a 12% increase in overall user adoption.

Role

Lead Designer

Team

Product manager, Engineers, and QA engineer

Duration

12 months

Background

In software quality assurance, performance testing is key to delivering fast and scalable software. Performance testing analyzes the speed, scalablity, stability, and responsiveness of an application or API under a particular workload.

Problem

We discovered early on QA engineers conduct performance tests to measure different types of metrics. The two main methods we identified is load testing and stress testing. Load testing helps developers understand the behavior of a system under a specific load value. While stress testing places a system under higher-than-expected traffic loads so developers can see how well the system works above its expected capacity limits. At mabl, we needed to design a new software product to meet these two core methods of performance testing and provide insightful metrics to our users.

UX Research

Competitor Research

To begin, we needed to define the requirements for performance testing. We conducted competitor research to understand the scope of settings and features.

After researching over six different competitors, we discovered that each software applied settings and load parameters at the test level. Each test is composed of multiple API requests or browser steps. The reporting of results can then be filtered by individual test steps. From this research, we outlined the test settings and reporting requirements.

Test settings:

  1. Add an API or Browser test
  2. Define load configuration (up to 1000 users) per test
  3. Define test failure criteria
  4. Add one or failure criteria for each test
  5. Set test duration (up to 1 hour)
  6. Set ramp up time

Reporting requirements:

  1. Response Time
  2. Concurrency
  3. Transaction Rate
  4. Throughput
  5. Failed Rate
  6. Pass Rate

User Interviews

We met with eight of our largest clients to understand their needs for performance testing. The users included of CEO's, engineering managers, and QA engineers. From our interviews we discovered:

    Key Findings:

  • 75% were in need of both Browsers & APIs load testing.
  • 80% wanted to see API endpoint and page URL level metrics in results.
  • 90% mentioned the importance of capturing a history of performance test runs to compare and evaluate metrics overtime.
  • All clients needed to be able to set one or more failure criteria for a single test.
  • All clients wanted to view and compare core web vitals for browser tests as they changed under different load configurations.

Strategize

From the research we gathered, the team and I began to brainstorm the short term and long term goals. We realized it would be a difficult implement both API and browser load testing at the same time. Based on the timeline, we decided to first design the ability to create API performance tests because that was where the most complex logic lived. Then in the second half of the year focus on building the addition of browser performance tests with captured core web vitals.

Prototype

Based on the key findings, I created a prototype to illustrate the information hierarchy, interactions, and metrics to display. I went through hundreds of iterations of the test creation form and results pages. After conducting user testing to evaluate the designs, I identfied usablilty issues and missing features. I made the following improvements:

Design Iterations

  • Add ablity to filter the graph by an individual test step.
  • Allow user to zoom into the graph to see the metrics captured in a specific timeframe.
  • Enable ability to add and remove metrics displayed on graph.
  • Give users the ability to display median metrics at the 50th, 75th and 90th percentile.

Adding Browser Test & Metrics

With the addition of browser loading testing, the team was faced with the new challenge to capture, store, and analyze browser test metrics. I conducted research online to understand the threshold scores of core web vitals captured. Core web vitals is a set of metrics that measure a web page's loading performance, interactivity, visual stability. This information was crucial in helping me design the failure criteria options for a browser test.
