Performance Testing

API and Browser Load Testing

Abstract

In 2023, mabl developed and launched a new product to allow users to create and run performance tests. I led the research, ideation, and design of the new product experience that result in a 12% increase in overall user adoption.

Overview

Background

In software quality assurance, performance testing is key to delivering fast and scable software. Performance testing analyzes the speed, scalablity, stability, and responsiveness of an application or API under a particular workload.

Problem

We discovered early on QA engineers conduct performance tests to measure different types of metrics. The two main methods we identified is load testing and stress testing. Load testing helps developers understand the behavior of a system under a specific load value. While stress testing places a system under higher-than-expected traffic loads so developers can see how well the system works above its expected capacity limits. At mabl, we needed to design a new software product to meet these two core methods of performance testing and provide insightful metrics to our users.

UX Research

Competitor Research

To begin, the product manager's and I had to define the scope of requirements for performance testing. We conducted competitor research to understand the scope of settings and features needed.

After researching over six different competitors, we discovered that each software tool applied settings and load parameters at the test level. Each test is composed of multiple API requests or browser steps. The reporting of results can then be filtered by individual test steps. From this research, we outlined the set of test settings and reporting requirements.

Test settings:

  1. Ability to add an API or Browser test
  2. Define load configuration (up to 1000 users) per test
  3. Define test failure criteria
  4. Add one or failure criteria for each test added
  5. Set test duration (up to 1 hour)
  6. Set ramp up time

Reporting requirements:

  1. Response Time
  2. Concurrency
  3. Transaction Rate
  4. Throughput
  5. Failed Rate
  6. Pass Rate

User Research

We met with eight of our largest clients to understand their organizations needs for performance testing. The individuals we interviewed included CEO's, engineering managers, and QA testers & engineers. From our interviews we discovered:

Key Findings

  1. Load testing for Browsers & APIs are both common use-cases, but API load testing was the higher priority.
  2. About 75% of customers found a greater need for API testing.
  3. API endpoint-level metrics and page-level URL performance is top of mind when viewing results.
  4. It is important to capture historical metrics overtime to measure performance and understand if you're gaining or losing ground.
  5. For browser loading testing, there is currently no way to assess how core web vitals change under load.

Strategize

From the research we gathered, the team and I began to brainstorm the short term and long term goals. We realized it would be a difficult implement and deliver both API and browser load testing at the same time. Based on the timeline, we decided to first design and build the ability to create API performance tests because that was where the most complex logic lived. In the second half of the year we would add browser tests with captured core web vitals.

Wireframes

Mistakes of early wireframes

From our early concepts, we worked on creating various wireframes to help us understand the information heirarchy and interactions of each deeper level metrics. We went through hundreds of iterations of the test creation form, test details, and results page. From our wireframes we conducted more user testing to identfied usablilty issues and points of confusion. This is a summary of things we discovered:

  1. There is no ablity to filter the graph by a individual test step.
  2. In the Response Time and Response Details tabs, at first glance is confusing to show historical test run metric information.
  3. Cannot zoom into graph to see the metrics captured in a specific timeframe.
  4. No functionality to add or remove metrics displayed on graph.
  5. It is better to display median metrics vs averages because the 50th, 75th and 90th percentile median give a more accurate representation of the application's performance.

Adding Browser Test & Metrics

With the addition of browser loading testing, the team was faced with the new challenge to capture, store, and analyze browser test metrics. I conducted research online to understand the thresholds scores of core web vitals captured by Google. Core web vitals is a set of metrics that measure a web page's loading performance, interactivity, visual stability. This information was crucial in helping me design the failure criteria options for a browser test.
