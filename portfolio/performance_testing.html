Performance Testing

API and Browser Load Testing

Overview

In 2023, mabl developed and launched a new product to allow users to create and run performance tests. I led the research, ideation, and design of the new product experience that result in a 32% increase in overall user adoption.

Role

Lead Designer

Team

Collaborated with business stakeholders, product managers, engineers, and QA engineers.

Duration

12 months

Final Prototype

Background

Mabl is a cloud-based test automation platform designed to assist companies in delivering high-quality software. Users of Mabl can construct robust end-to-end tests for their software applications. These tests can be executed across browsers, APIs, and mobile devices to identify functionality issues.

Problem

Within the mabl platform, users could not run load tests to assess their application's speed, stability, and responsiveness. This limitation prevented software teams from identifying performance issues before deploying new code to production. Users of mabl expressed a desire to leverage their existing API and browser tests for load testing to facilitate reuse.

Objective

The objective was to expand mabl's testing capabilities by designing a comprehensive performance testing tool that integrates both API and browser load testing. This new tool aims to provide accurate metrics and insights for teams to identify bottlenecks in web-based applications.

Design Process

UX Research

Competitor Research

To begin, we needed to define the requirements for performance testing. We conducted competitor research to understand the scope of settings and features.

After researching over eight different competitors, we discovered that each software applied settings and load parameters at the test level. Each test is composed of multiple API requests or browser steps. The reporting of results can then be filtered by individual test steps. From this research, we outlined the test settings and reporting requirements.

Test settings:

  1. Add an API or Browser test
  2. Define load configuration (up to 1000 users) per test
  3. Define test failure criteria
  4. Add one or failure criteria for each test
  5. Set test duration (up to 1 hour)
  6. Set ramp up time

Reporting requirements:

  1. Response Time
  2. Concurrency
  3. Transaction Rate
  4. Throughput
  5. Failed Rate
  6. Pass Rate

User Interviews

We met with twelve of our largest clients to understand their needs for performance testing. The users consisted of CEOs, engineering managers, software and QA engineers. From our interviews we discovered:

    Key Findings:

  • 75% were in need of both Browsers & APIs load testing.
  • 80% wanted to see API endpoint and page URL level metrics in results.
  • 90% mentioned the importance of capturing a history of performance test runs to compare and evaluate metrics overtime.
  • All clients needed to be able to set one or more failure criteria for a single test.
  • All clients wanted to view and compare core web vitals for browser tests as they changed under different load configurations.

Core User Persona

After concluding our initial research, we pinpointed our core user persona: QA Engineers. Their primary objectives revolved around identifying performance bottlenecks, measuring scalability, and ensuring reliability and stability for end-users.

Strategize

Based on the research we gathered, we brainstormed both short-term and long-term goals to establish a roadmap for the year.

Based on our timeline, we decided to first design and implement API performance testing.
In the second half of the year, our focus shifted towards incorporating browser load tests with captured Core Web Vitals.

Initial Prototype

From our research and key findings, I developed an initial prototype to visualize the information architecture, interactions, and metrics to captured. I iterated through hundreds of versions of the test creation form and results pages to ensure alignment with user needs and expectations. The team decided to move forward and construct the initial prototype for use in our early access beta phase.

API Beta (Early Access)

We launched API performance testing for early access to our customer base in April 2023. However, in the initial months, we observed a low adoption rate, with less than 10% uptake.

Based on the low adoption rates, I revisted my initial designs and conducted a second round of user testing to evaluate its effectiveness. During my process, I identfied usablilty issues and key missing features. I made the following improvements:

Design Iterations

  • Add ablity to filter the graph by an individual test step.
  • Allow user to zoom into the graph to see the metrics captured in a specific timeframe.
  • Enable ability to add and remove metrics displayed on graph.
  • Empower users with the capability to showcase metrics at the 50th (median), 75th, and 90th percentiles.
